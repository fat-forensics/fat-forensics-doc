Ideas List – Google Summer of Code (GSoC) 2020¶

Useful Information¶

Before choosing a project from the project list or proposing your own project, please have a look at the Development Roadmap of FAT Forensics.

A collection of GitHub Issues relevant to our GSoC projects can be found here; these are tagged with a GSoC label and will appear over time.

If as part of your application you wish to develop a prototype of a certain feature or submit a pull request, please make sure to read thorough our Developers Guide. In particular, you should make a note of our Code Formatting, which will help you adhere to our coding style.

If you want to discuss a project with us, please reach out via dedicated GSoC room on gitter.

We look forward to receiving your applications!

Application Template¶

When planning your project please make sure that you take into consideration the time needed for code review and merging your contributions into upstream. Writing code documentation, tests and its appropriate formatting should be done alongside the code development and the work plan should reflect that. A complete but small contribution is valued more than a lot of functionality without tests and documentation since the latter cannot be merged without additional work.

As a part of your final report summarising your work during GSoC, we will ask you to write a How-to Guide(s) for the feature(s) that you developed. As a reference you may want to have a look at one of the existing guides: How to build LIME yourself (bLIMEy) – Surrogate Tabular Explainers.

Please provide the following information at the top¶

  1. full name,

  2. email address,

  3. phone number (inc. country code),

  4. GitHub profile URL,

  5. nationality & mother tongue,

  6. country from which you will be working (inc. time zone),

  7. university name, course details, graduation date,

  8. how many GSoC applications you have sent this year and the priority our project.

Please answer the following questions¶

  • Are you proposing your own project or applying for one from our list?

  • Why do you want to work on FAT Forensics and why did you choose this particular project?

  • Are you contributing to open source projects (indicate names) on a regular basis?

  • Have you participated in GSoC before; what was the organisation/project, the results and where to find them?

  • What is your background in Artificial Intelligence, Machine Learning, algorithmic Fairness/Accountability/Transparency and Python programming?

  • Do you have any relevant skills and experience?

  • Why should we choose you?

  • Can you see anything in FAT Forensics that is relevant to your proposed project and can be improved (open an issue and provide a link)?

  • Have you submitted any PR addressing issues highlighted in the project list (provide links)?

  • Do you plan holidays or have any other commitments over the summer?

Proposal content¶

Please discuss your project details supported by an outline of envisaged timeline/work plan for each week of GSoC.

Please note that we will give preference to students who select high priority projects and who submit a PR as outlined above.

Projects¶

One of the FAT Forensics’ goals is to provide a modular implementation of relevant functionality alongside a high-level API targeted at the end-users. If you propose to implement a particular tool or technique, please give some thought to its implementation design, focusing on independent algorithmic components that could be re-used for other implementations. For example, a surrogate explainer can be composed from a data augmenter, interpretable representation and local model; this modular approach is described in the How to build LIME yourself (bLIMEy) – Surrogate Tabular Explainers How-to Guide.

Note

Skills and Experience

All of the projects will require you to write Python code, document it – using reStructured Text (reST) – and write appropriate unit and integration tests (PyTest). A background in Machine Learning or Data Science is essential. Knowledge of algorithmic fairness, accountability and transparency is a plus, however not required.

Project 1: Feature Importance and Influence¶

Title

Feature Importance and Influence

Tag

Transparency

Difficulty

Medium

Priority

High

Extra Technology

N/A

Mentor

Alex Hepburn

Note

Motivation

Working on this project you will implement various algorithms to help the user better understand how a black-box predictive model, e.g. a machine learning algorithm, is influenced by features in the training data. This set of tools will help to debug and explain opaque models, hence engender confidence in their usage and deployment.

There are two types of feature influence and importance that need to be considered: global and local. The first one reflects how features influence the behaviour of the model as a whole; the latter indicates how features contribute to a prediction of a particular data point. Throughout the project you will investigate these techniques and implement them within FAT Forensics. Some of them are listed in the Development Roadmap; others are discussed in blog posts such as this one. A successful project will conclude in merging tested and documented code into upstream and writing a relevant User Guide and/or How-to Guide.

If you’re familiar with the topic and want to drive this project an extra mile, you can consider implementing alternative transparency tools that also use features as the explanations, but in a different way. For example, ANCHORS indicate which features should be kept fixed to maintain the current prediction given by a black-box model.

Project 2: Influential Data Points¶

Title

Influential Data Points

Tag

Transparency

Difficulty

Hard

Priority

High

Extra Technology

Python’s multiprocessing, Cython (optional)

Mentor

Kacper Sokol

Note

Motivation

Working on this project you will implement various algorithms aimed at explaining predictions of black-box models using exemplars, i.e., relevant data points. These techniques are particularly useful for audience who is not familiar with Machine Learning concepts, therefore requires “explanations by example”.

With this project we hope to implement exemplar-based explanations for predictions of black-box Machine Learning models. Ideally, some of these tools would be model-agnostic – applicable to any predictive model – however model-specific approaches – designated for a particular model class, e.g., decision trees – could be a good start and constitute a base of a successful project. Such techniques can be found under different names, e.g., influential examples, prototypes and criticisms or influence functions. ProtoDash and contrastive explanations – also described in this paper – may be considered as well.

Since some of these techniques may require significant computation, a savvy student may decide to parallelise the code or optimise it with Cython. This demanding task can either be woven into the original plan or executed once the core deliverable of the project has been achieved.

Project 3: Measuring and Mitigating Unfairness¶

Title

Measuring and Mitigating Unfairness

Tag

Fairness

Difficulty

Easy/Medium

Priority

Medium

Extra Technology

N/A

Mentor

Raul Santos-Rodriguez

Note

Motivation

Predictive models often come with unintended biases that can lead to discrimination of groups and individuals underrepresented in the training data set, e.g., for historical reasons. Working on this project you will implement a pipeline for detecting, measuring and mitigating unfair behaviour of machine learning models.

This project will require you to discover the fast-paced landscape of fairness in predictive modeling and implement state-of-the-art techniques for detecting, measuring and mitigating algorithmic bias. A successful project will have implemented a proof-of-concept pipeline consisting of at least one algorithm for each of these tasks. A fairness and machine learning book may be a useful background reading material if you are new to these concepts and perusing the source code of the AI Fairness 360 software package may be a good technical introduction.

Project 4: Classifier Calibration for Reliable Probabilities¶

Title

Classifier Calibration for Reliable Probabilities

Tag

Accountability

Difficulty

Hard

Priority

Low

Extra Technology

N/A

Mentor

Peter Flach

Note

Motivation

Probabilistic predictive models often output uncalibrated predictions – the estimated probability does not reflect the probability of the underlying event. Such behaviour leads to over- or under-confident predictions, which may harm the users’ trust in the system. While working on this project you will implement state-of-the-art calibration techniques that can “fix” output of such classifiers, engendering trust in their usage.

In this project you will implement state-of-the-art calibration techniques as part of FAT Forensics accountability module. While scikit-learn takes care of basic calibration algorithms, these are far from the bleeding edge. Novel approaches such as beta calibration, distribution calibration or Dirichlet calibration. A successful project would implement at least one of these techniques as a model-agnostic tool.
